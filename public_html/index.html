Project Specification

Hivesense is a project to enrich the process of monitoring beehives, using off-the-shelf hardware and free, open-source software. This web application, along with the device that produces the data feed, have been developed as part of a Masters' project at UCL running between June and August 2013. The blog linked on the left gives more details. The main aim of this project is to develop a customisable software stack that can be easily implemented by beekeepers across the land, and furthered by future projects. I welcome feedback and suggestions of any nature (see email in the footer).

Details

Hardware

Variety of sensors to measure environmental properties, a camera, and a motion detector (alert when hive knocked-over).
Full parts List
  • GHI FEZ mainboard with power and Wi-Fi modules, plus temperature/humidity, accelerometer, barometer/temperature, and light sensors (Source)
  • SD card for loading configuration (Wi-Fi settings, server URL etc.)
  • Power supply (anything in range 5-30V)
  • Mounting kit - prototyping plate plus assorted ironmongery
  • Other Gadgeteer sensors - as desired.
Total cost: ~Â£200
Installation time: ~1hr.

Monitoring device which produces the data. Dimensions: 16 x 12 cm

Software

Gadgeteer SDK, pushed to Node.js server hosted on Azure, sensor data stored in Azure Tables database, JS front-end (all on GitHub, along with all dependencies, so web part is directly deployable to any Node 0.8+ compatible environment).
Requirements: Windows Azure account.

HiveSense API

Retrieving data

Data can be freely obtained directly from the Hivesense API by anyone

Base URL
http://hivesensenodejs.azurewebsites.net/feed
HTTP Method
GET
Caching
Results are cached for the period of the sensor update rate (approx. every minute, but configurable)
Format
Results can be obtained in one of the three possible formats: .json (default), .xml, .csv
Response
Ordering
Descending; when multiple datpoints are returned, the most recent will be at the top.
Client Errors
A 400 error will be returned if no data is available for the specified period or if a malformed query is sent. The response body will always be in JSON and contain a short descriptive message.
A general note on timestamps
These must be UTC and in a Javascript-friendly format: millisecond Unix time or an ISO date-string. Returned data uses Unix time for optimal brevity, or ISO strings when a readable format is desirable.

Current Data

URL ?current
Returns Most recent single datapoint
Example /feed.xml?current

Current Hivecam

URL ?hivecam
Returns Most recent available camera image (bmp format)
Sample .../feed?hivecam

Recent Data

URL ?recent=[options]   where options is any number followed by the period-type: h,d or m (hours, days, months)
Returns Most recent data available for the period specified, suitable sampled (somewhere between the sensor update rate and one day) to limit the response to 1000 datapoints.
Example .../feed.json?recent=0.5h

Historical Data

URL ?date1=[timestamp]&date2=[timestamp]   where timestamp is as described above.
Returns Suitably sampled data from the specified range.
Example /feed.csv?date1=02 August 2013 13:42Z&date2=1375358405000

Inputting data

This should be handled by the monitoring device itself, but these docs provide usage for manual testing, which can be useful.

Base URL
http://hivesensenodejs.azurewebsites.net/posty
Method
PUT or POST (no difference at present).
Format
Only JSON is supported at this time (for datapoints), all other formats will be rejected.
NB: There is a special exception for sending binary data through the dedicated pathway /posty?image
Security
Only users with a valid "X-HiveSenseSecureKey" header will be successfully authenticated (receiving a 200 response), others will be rejected with a 401 Unauthorised.
Caching
Results are never cached.
Response body
A 200 response will return a short JSON success message, a 401 response will include a descriptive JSON error.

Single Datapoint

{
  "datapoints": [
    {
      "channels": {
        "temp": "34.6",
        "humi": "54",
        "light": "0",
        "key": "value"
      }
    }
  ]
}

Multiple Datapoints

{
  "datapoints": [
    {
      "channels": {
        "temp": "37.2",
        "humi": "52",
        "light": "0",
        "key": "value"
      },
      "datetime": "2013-07-21 12:00:02 Z"
    },
    {
      "channels": {
        "temp": "35.2",
        "humi": "42",
        "light": "5",
        "key": "value"
      },
      "datetime": "Javascript-compatible timestamp"
    }
  ]
}

Image

Use the special pathway /posty?image and send a Content-Type: application/octet-stream HTTP header
